# Eurovision Voting – Software Architecture Specification

> No implementation code. Structure, interfaces, responsibilities, and integration rules only.
> All previous design decisions (removed fromCountryId, NULL-constraint handling, Locale fix) are reflected here.

---

## 1. GLOBAL ARCHITECTURE

### Layer Definitions

```
FRONTEND  (HTML / CSS / JavaScript)
  Communicates via fetch() only. No business logic.
  Reads USE_MOCK flag to switch between mock and real API.
         |
         | HTTP / JSON
         v
CONTROLLER LAYER  (@RestController)
  Receives HTTP requests. Validates input format (@Valid).
  Delegates to Service. Returns HTTP responses.
  Contains: ZERO business logic.
         |
         | Java method calls
         v
SERVICE LAYER  (@Service)
  Contains: ALL business logic. ALL validation rules.
  Calls Repository. Maps Entities to DTOs.
  Never returns Entity objects directly to Controller.
         |
         | Java method calls
         v
REPOSITORY LAYER  (JpaRepository interfaces)
  Database access only. Named queries + JPQL.
  Contains: ZERO logic. ZERO validation.
         |
         | JPA / SQL
         v
DATABASE  (H2 in-memory)
  Schema auto-generated by Hibernate (ddl-auto=create-drop).
```

### Naming Conventions

| Element | Convention | Example |
|---------|-----------|---------|
| Package root | com.eurovoting | |
| Entity classes | PascalCase, singular | Country, Vote |
| DTO classes | PascalCase + suffix | VoteRequest, VoteResponse, RankingEntry |
| Service classes | PascalCase + Service | VoteService, ResultService |
| Controller classes | PascalCase + Controller | VoteController |
| Repository interfaces | PascalCase + Repository | VoteRepository |
| Exception classes | PascalCase + Exception | VotingException |
| Enum values | SCREAMING_SNAKE_CASE | JURY, AUDIENCE |
| JSON field names | camelCase | artistName, juryMemberId |
| DB column names | snake_case (auto by JPA) | jury_member_id, our_country |

### Endpoint Overview

| Method | URL | Owner | Purpose |
|--------|-----|-------|---------|
| GET | /api/contestants | Person 3 | List all contestants |
| POST | /api/votes | Person 1 | Cast a vote |
| GET | /api/votes/status | Person 1 | Voting progress for our country |
| GET | /api/results | Person 2 | Aggregated ranking |

Base URL: http://localhost:8080/api
Content-Type: application/json on all requests and responses.
CORS: @CrossOrigin(origins = "*") on all controllers (dev only).

| DataInitializer | @Component | com.eurovoting.config |

Must provide these method signatures:

```
CountryRepository:
  findByOurCountryTrue()  :  Optional<Country>
  findBySimulatedTrue()   :  List<Country>

ContestantRepository:
  findAll()               :  List<Contestant>  (inherited from JpaRepository)

VoteRepository:
  existsByFromCountryAndVoterRoleAndJuryMemberIdAndPoints(
      Country, VoterRole, Integer, int)         :  boolean
  existsByFromCountryAndVoterRoleAndJuryMemberIdAndToContestant(
      Country, VoterRole, Integer, Contestant)  :  boolean
  sumPointsByContestantAndRole(
      Contestant contestant, VoterRole role)    :  int   (JPQL @Query, COALESCE to 0)

  GET /api/contestants -> getAllContestants()   :  ResponseEntity<List<ContestantResponse>>
```

Important rules:
- DataInitializer inserts countries + contestants ONLY. Zero votes.
- Exactly ONE country must have ourCountry = true. Minimum 11 countries total.
- All VoteRepository method names must match exactly as specified above.
- countVotesByVoter must accept Integer memberId as nullable (null for AUDIENCE).
- sumPointsByContestantAndRole must use COALESCE(SUM(...), 0) - never return null.

Depends on: Nobody. Starts first.
Others depend on this: Person 1 and Person 2 cannot start without this.

### Person 1 - Vote Submission

Responsibility: Complete vote submission pipeline including all validation,
duplicate prevention, and voting status endpoint.
Also owns the central error handling used by all controllers.

Must create these classes:

| Class | Type | Package |
|-------|------|---------|
| VotingException | RuntimeException | com.eurovoting.exception |
| NotFoundException | RuntimeException | com.eurovoting.exception |
| GlobalExceptionHandler | @RestControllerAdvice | com.eurovoting.exception |
| VoteRequest | DTO class | com.eurovoting.dto |
| VoteResponse | DTO class | com.eurovoting.dto |
| VotingStatusResponse | DTO class | com.eurovoting.dto |
| VoteService | @Service | com.eurovoting.service |
| VotingStatusService | @Service | com.eurovoting.service |
| VoteController | @RestController | com.eurovoting.controller |

Must provide these method signatures:

```
VoteService:
  castVote(VoteRequest request)     :  VoteResponse

VotingStatusService:
  getStatusForOurCountry()          :  VotingStatusResponse

VoteController:
  POST /api/votes
    -> castVote(@Valid @RequestBody VoteRequest)
                                    :  ResponseEntity<VoteResponse>          [HTTP 201]
  GET /api/votes/status
    -> getStatus()
                                    :  ResponseEntity<VotingStatusResponse>  [HTTP 200]

GlobalExceptionHandler:
  handleVoting(VotingException)                    :  ResponseEntity<Map<String,Object>>
  handleNotFound(NotFoundException)                :  ResponseEntity<Map<String,Object>>
  handleValidation(MethodArgumentNotValidException):  ResponseEntity<Map<String,Object>>
  handleGeneric(Exception)                         :  ResponseEntity<Map<String,Object>>
```

Important rules:
- VoteService.castVote() calls countryRepository.findByOurCountryTrue() internally.
  fromCountryId does NOT exist in VoteRequest.
- Validation order inside castVote() must be exactly:
    1. Validate points in {1,2,3,4,5,6,7,8,10,12}  ->  400
    2. Validate no self-voting                       ->  409
    3. Validate juryMemberId consistency             ->  400
    4. Validate no duplicate points for this voter   ->  409
    5. Validate no duplicate contestant for this voter -> 409
- Rules 4 and 5 must call VoteRepository methods directly.
  They must NOT rely on DB constraint alone (DB constraint fails for null juryMemberId).
- VotingException: one constructor with message only (defaults HTTP 409),
  one with message + HttpStatus.
- GlobalExceptionHandler error response format is frozen (see Section 2).

Depends on: Person 3 (entities, repositories, findByOurCountryTrue).
Others depend on this: Person 4 cannot switch USE_MOCK = false until this is complete.

### Person 2 - Result Aggregation

Responsibility: Calculate and serve the final ranking. One service, one controller, one DTO.

Must create these classes:

| Class | Type | Package |
|-------|------|---------|
| RankingEntry | DTO class | com.eurovoting.dto |
| ResultService | @Service | com.eurovoting.service |
| ResultController | @RestController | com.eurovoting.controller |

Must provide these method signatures:

```
ResultService:
  calculateRanking()     :  List<RankingEntry>

ResultController:
  GET /api/results
    -> getResults()      :  ResponseEntity<List<RankingEntry>>   [HTTP 200]
```

Important rules:
- calculateRanking() fetches ALL contestants. Every contestant appears, even with 0 points.
- For each contestant: call VoteRepository.sumPointsByContestantAndRole() twice
  (once for JURY, once for AUDIENCE).
- totalPoints must ALWAYS be computed as juryPoints + audiencePoints in Java code.
  Never stored or fetched separately.
- Sorting: primary totalPoints DESC,
           secondary countryName.toLowerCase(Locale.ROOT) ASC.
  Locale.ROOT is mandatory to avoid case and locale-dependent sorting bugs.
- rank is assigned after sorting, starting at 1, no gaps.
- With no votes: returns all contestants sorted alphabetically with 0 points.
  Never returns null or empty array.

Depends on: Person 3 (entities, ContestantRepository, VoteRepository).
Others depend on this: Person 5 cannot switch USE_MOCK = false until this is complete.

## 5. INTEGRATION RULES

### Rule 1 - JSON Field Names Are Frozen

These field names must never change after document distribution:

```
VoteRequest:        toContestantId, voterRole, points, juryMemberId
VoteResponse:       id, fromCountryName, toArtistName, toSongTitle,
                    toCountryName, voterRole, points, juryMemberId, createdAt
ContestantResponse: id, artistName, songTitle, countryName, countryCode
RankingEntry:       rank, contestantId, artistName, songTitle, countryName,
                    countryCode, totalPoints, juryPoints, audiencePoints
ErrorResponse:      status, error, message, timestamp
```

### Rule 2 - Null Handling Is Explicit

- juryMemberId in VoteRequest: must be present in JSON even as null. Never omit.
- juryMemberId in VoteResponse: present as null for AUDIENCE votes.
- rank, juryPoints, audiencePoints, totalPoints in RankingEntry: never null. Use 0 as minimum.
- Contestant lists: never null. Empty array [] when no data.

### Rule 3 - Error Format Is Universal

Every HTTP error from every endpoint must use the frozen ErrorResponse structure.
Frontend reads error.message to show to the user.
If format changes, all frontend error handling breaks simultaneously.

### Rule 4 - fromCountryId Does Not Exist

The field fromCountryId must not appear anywhere in VoteRequest.
Backend determines fromCountry by findByOurCountryTrue().
Any frontend code sending fromCountryId will be silently ignored or cause errors.

### Rule 5 - The DB Constraint Is Not Enough

The unique constraint on Vote does not fire when juryMemberId is null (AUDIENCE votes).
This is a known SQL behavior. VoteService is the single source of truth for duplicate
prevention. No other class may assume the DB handles this.

### Rule 6 - voterRole Always Uppercase

VoterRole enum uses STRING serialization strategy.
Always "JURY" or "AUDIENCE" in JSON. Frontend must never send lowercase.

### Rule 7 - Results Are Pre-Sorted

GET /api/results always returns data sorted by totalPoints DESC.
Frontend must display the array in the received order. Frontend must never re-sort.

### Rule 8 - Script Load Order

In both index.html and results.html, scripts must load in this exact order:
  1. api.js
  2. voting.js (or results.js)

If api.js loads after the page script, all function calls fail silently.
